# Levels of autonomy for AI

## Level 1: Chat (Training Knowledge)
The model responds purely from what it learned during training. No access to your data, no real-time updates. Ask about something outside its training set and you'll either get a hallucination or a straightforward "I don't know" — depending on how lucky you are.
Think: ChatGPT without plugins, Claude without projects, Gemini with no added context.

## Level 2: Structured Output (A Contract with the Model)
This level isn't about feeding the model new knowledge — it's about owning the output format. No more walls of text. Instead, you get clean, predictable structure: JSON, Python objects, tables.
It sounds simple, but it's non-negotiable. Level 3 (RAG) is a multi-component pipeline where pieces constantly hand off data to each other. If the model can't reliably return structured output at Level 2, the whole thing falls apart upstream.

## Level 3: RAG (External Knowledge) 
The model gets **access to your knowledge** — documents, databases, files, wikis. But here's the key insight: **RAG is not a single technique**. It's a whole family of approaches:

| Variant | Where the knowledge lives | How it's retrieved |
|---|---|---|
| Naive RAG | Chunks in a vector database | Cosine similarity |
| BM25 / Classic Search | Documents + inverted index | Keywords, TF-IDF |
| Structured Retrieval | Database / tables | SQL, metadata filters |
| Graph RAG | Entity and relation graph | Graph traversal |
| **LLM Wiki** (Karpathy) | Markdown files | LLM reads an index and selects pages |
| **Knowledge Map** | Structured artifacts with metadata | Multi-stage retriever: filters + LLM-based selection |

## Key Concepts in RAG Retrieval


### 1. Cosine Similarity
A mathematical measure of how similar two vectors are, based on the angle between them.
In RAG, both your query and document chunks are converted into vectors (lists of numbers).
Cosine similarity answers: *"Do these two vectors point in the same direction?"*
- Score of **1.0** → identical meaning
- Score of **0.0** → completely unrelated

It captures **semantic similarity** — so "car" and "automobile" score high,
even though they share no common words.

### The Math: cos(θ)

The **cosine** of the angle between two vectors gives the score:

| Angle (θ) | cos(θ) | Meaning              |
|-----------|--------|----------------------|
| 0°        | 1.0    | Identical direction  |
| 45°       | 0.71   | Somewhat similar     |
| 90°       | 0.0    | Completely unrelated |
| 180°      | -1.0   | Opposite meaning     |

### 2. Classic Search (BM25)
The traditional keyword-based search used by search engines for decades.
It looks for **exact or near-exact word matches** between your query and documents.
No understanding of meaning — just counting and ranking word occurrences.

### 3. Inverted Index
The data structure that makes classic search fast.
Instead of scanning every document for a word, it pre-builds a lookup table:
"neural"   → [doc_3, doc_7, doc_12]
"network"  → [doc_1, doc_3, doc_9]
"python"   → [doc_2, doc_5, doc_7]

When you search "neural network", it instantly fetches the intersection of those lists.
Think of it as the index at the back of a textbook — but for an entire database.

### 4. TF-IDF (Term Frequency – Inverse Document Frequency)
A scoring formula used in classic search to rank how *relevant* a word is:

- **TF (Term Frequency):** How often does the word appear in *this* document?
- **IDF (Inverse Document Frequency):** How rare is the word *across all documents*?

The insight: a word that appears often in one document but rarely elsewhere
is a strong signal. A word like "the" appears everywhere — it carries no signal.

**TF-IDF = common in this doc × rare across all docs**

### 5. Graph RAG — How Is It Different?

All other RAG variants treat knowledge as **isolated chunks or documents**.
Graph RAG treats knowledge as a **network of connected entities and relationships**.

Instead of storing chunks, you build a graph:
[Python] ──used_in──> [Machine Learning]
[PyTorch] ──is_a──> [ML Framework]
[Andrej Karpathy] ──created──> [micrograd]

When you query, instead of finding the most similar chunk,
you **navigate the graph** — following relationships to gather connected context.

**Why it matters:**
- Classic RAG struggles with multi-hop questions:
  *"What frameworks does the person who created micrograd recommend?"*
- Graph RAG can traverse: `Karpathy → created → micrograd → recommends → PyTorch`
- It captures **relational context** that flat chunks simply can't express

> **RAG is not a tool — it's a mindset. System quality lives or dies in two places:
> how you prepared your knowledge, and how you retrieve it.**

## Level 4–5: Memory, Tools, Planning

Level 4 is an agent with memory and tools. Level 5 is an agent that can plan on its own.

**The catch:** a smart agent is useless without good knowledge behind it.
It needs to get information from somewhere — and that's exactly what RAG is for.

## What RAG Actually Is (and Isn't)

### Myth #1: "RAG = vector database"
Wrong. A vector database is just one retrieval tool — like saying "a car is an engine."
When Anthropic dropped vector search in Claude Code, they didn't drop RAG.
They still retrieve — using files, grep, content search, and an LLM deciding what to read.

### Myth #2: "Karpathy showed something revolutionary"
Not quite. His core idea:

> *"Don't feed raw documents into the retriever. Compile them first into structured form (a wiki).
> Then retrieval becomes easy — because you're searching well-organized knowledge."*

New? No. But did the market need someone like Karpathy to say it out loud
to move on from "vector DB solves everything"? Absolutely.


### So What Is RAG, Really?

RAG is a system built from at least four layers:

| Layer | Details |
|---|---|
| **Ingestion** | Document parsing · Format normalization · Structure extraction (tables, sections) |
| **Knowledge Representation** | Structured artifacts · LLM-generated metadata · Concept relationships · Human-auditable and editable |
| **Retrieval** | Metadata filtering · LLM selects the right artifacts · BM25, embeddings, graph — pick your tool |
| **Generation** | Context + question → LLM · Structured answers with citations · Confidence scores, sources, audit trail |

See where the real work happens? **Knowledge representation.**
That's the layer most people skip — and why most RAG systems underperform.

### What Is an Artifact?

An artifact is a processed, structured unit of knowledge — not a raw document.

| | Raw Document | Artifact |
|---|---|---|
| Form | 40-page PDF | Clean, structured object |
| Metadata | None | Title, tags, type, source, page... |
| Retriever-friendly? | ❌ | ✅ |

Artifacts are the output of the **Knowledge Representation** layer —
where raw content gets transformed into something a retriever can filter and select intelligently.

### LLM-Generated Metadata

You run the raw document through an LLM **before** storing it, and ask it to extract structured tags automatically:

```json
{
  "title": "How to reset the device",
  "type": "procedure",
  "topic": "troubleshooting",
  "product": "SensorUnit v2",
  "difficulty": "beginner",
  "requires": ["admin access", "physical device"],
  "summary": "Step-by-step guide to factory reset via the control panel"
}
```

You didn't write any of that manually — the LLM read the chunk and generated it.

**Why it matters:** without metadata, the retriever is blind — it can only match on raw text similarity.
With it, you can filter like a database:
type = "procedure"
AND product = "SensorUnit v2"
AND topic = "troubleshooting"

That's orders of magnitude more precise than cosine similarity alone.
